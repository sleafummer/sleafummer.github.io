认识一些核心概念

1 ）JSX ： 相当于 HTML 模板 ， 目标是渲染 DOM 。

对初学者来说 ， 会有为什么 React 会搞出 JSX 这个屎一样的东西的疑问 。

React 是这么解释的 ， 他认为页面的渲染逻辑本质上与其他 UI 逻辑内在耦合 。 比如 ， 在 UI 中需要绑定事件 ， 在某些状态发生变化时需要通知到 UI , 以及需要在 UI 中展示数据 。

然而 ， 这并不能说服我 ， 他本身就是通过工具来实现这种 JSX ， 完全可以人为的将 JS 、 CSS、 HTML 分离到不同的文件当中。但是 ， 我看了他的视频 ， 我又觉得他貌似有有点道理 ， 他说 ， 组件不是模板 。 这点我非常认同 。 但是但是但是 ， 我们都喜欢分离 ， 不是么 。

当然 ， React 也说他不强制使用 JSX ， 你可以看 React Without JSX 。 当然我肯定不会现在看的 。

最后 ， 他还不要脸的说 ： 在 JavaScript 代码中将 JSX 和 UI 放在一起时，会在视觉上有辅助作用 。 它还可以使 React 显示更多有用的错误和警告消息 。

2 ）虚拟DOM

React 只更新它需要更新的部分
它利用的是一个叫虚拟 DOM 的东西 ， 总之 ， 现在知道就行了 ， 我肯定不会现在看他的 。

3 ）组件

组件分为2种 ， 一种叫 函数组件 ， 另一种叫 React 组件 。 这两种组件在 React 里是等效的 。

所有 React 组件必须是纯函数

4 ）ReactDOM

ReactDOM 负责将组件挂载到页面上 。

5 ）props

组件通过 props 来接收数据 ， 对于组件来说 ， props 是只读的 ， 组件不能修改自身的 props 。 因为 React 组件必须设计成为纯函数 。 不能更改自己的入参 。

6 ）state

与 props 相对应 ， 但是 state 是组件私有的 ， 并且只受当前组件控制 。 也就是说组件可以动态更改 state 。

7 ）生命周期

任何框架都有生命周期 ， 实际上就是一系列的函数钩子 ， React 的生命周期是基于组件的 。 也就是说 ， 每个组件都有一套自己的生命周期 。

基本上可以归纳为以下几个 ：

  1. 组件初始化
  2. 组件挂载
  3. 组件更新
  4. 组件卸载

8 ）事件处理

基本上跟 DOM 事件处理 差不多 ， 只是有些语法上的差异 。

有一点非常重要 ， 就是必须谨慎对待 JSX 回调函数中的 this ， 如果不注意 ， 会引起 bug 。

通常在构造函数中用剪头函数就能解决啦 。

事件传参 ， 这个也需要注意 。

9 ）Keys

在 React 中 ， keys 比较重要 ， 因为 Diff 算法需要用到他 。
当元素没有 id 的时候 ， 万不得已 ， 你可以使用元素的索引作为 key 。
如果列表的顺序可能发生变化 ， 千万不要用 index 作为 key ，这是极不负责任的 。

10 ）表单

为什么会吧表单也作为 React 的核心概念来讲 。

因为在 React 中 ， 可变的数据状态都保存在组件自身的 state 中 ， 并且只能通过 setState 来更新 。

但是 ， 像 input select textarea 这些元素是自己维护自己的数据的 ， 并根据用户的输入进行更新 。

所以 ， React 需要对这些表单元素进行额外的控制 ， 你们不能维护自己的数据 ， 你们只能使用组件的 state 作为自身唯一的数据源 。 如果我 state 不变 ， 即使用户输入了什么 ， 你也不能变 。

最后 ， 引出一个概念 。 受控组件 。

怎么控制呢 ？ 大致像下面这样做 。

<input value={this.state.shopName} />

其实 ， 我觉得也没有必要弄出一个受控组件的概念 。 感觉是多余的 。可能我在后面会改变这种想法 ， 但至少我现在是这样人为的 。

当然 ， 有受控组件 ， 也有非受控组件 。这是实现输入表单的另一种方式 。 当然 ， 我肯定不会现在看它的 。

大多数情况下 ， React 推荐使用受控组件 。

结尾

写到这里 ， 我大概已经知道 React 是什么了 。 它使用组件的方式渲染 UI 界面 ， 并提供了一些数据 、 钩子函数 、 事件的一些机制 。

接下来 ， 我会用纯 React 实现一个真实 API 的分页 TodoList ， 马上进入 React 学习之旅吧 。
